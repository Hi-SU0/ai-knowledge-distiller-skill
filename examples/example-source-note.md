# 用检索增强生成回答内部资料问题

> 本文件使用虚构来源演示格式，不代表真实视频内容。

## 一句话摘要

视频用一个内部文档问答案例解释 [[RAG]] 如何把检索结果作为上下文交给 [[LLM]] 生成可追溯回答。

## 来源要解决的问题

当模型原有知识不包含组织内部资料时，如何在不重新训练模型的情况下，让回答引用受控知识源？

## 目标受众与前置知识

面向了解基本 API 调用的应用开发者；建议先理解 [[LLM]]、[[Embedding]] 和 [[向量数据库]]。

## 内容地图

1. 问题边界（00:00）
2. 文档切分和索引（03:10）
3. 检索与生成流程（08:20）
4. 失败案例和评估（14:40）

## 核心结论

- `来源明确陈述`（03:10）：索引前需要将长文档切分为可检索片段。
- `来源明确陈述`（08:20）：检索结果需要与问题一起进入模型上下文。
- `创作者个人观点`（14:40）：创作者认为先改进检索质量通常比更换大模型更值得排查。
- `合理推导`：如果检索阶段没有召回证据，生成阶段很难稳定补救，因为模型没有获得目标资料。

## 论证与讲解过程

来源先界定模型知识缺口，再演示“准备文档—建立索引—检索片段—拼接上下文—生成回答—返回引用”的链路，并用一次召回失败说明评估不能只看答案是否流畅。

## 案例、类比与关键画面

`来源明确陈述`（10:05）：演示界面同时展示用户问题、召回片段和最终答案，便于定位错误发生在检索还是生成阶段。

## 实践方法

建立一个最小文档集，为每个测试问题人工标注期望证据；分别记录召回是否命中和答案是否忠实于证据。

## 容易误解的地方

RAG 不保证答案正确。它只提供外部上下文，仍需检查召回质量、上下文组织和生成忠实度。

## 创作者观点

创作者偏好先用可观察的检索指标排查系统，再调整生成提示词。

## 我的理解

这种排查顺序的价值在于把端到端问题拆成两个可测试阶段，但不同任务仍可能需要联合评估。
